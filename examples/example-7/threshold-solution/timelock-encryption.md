# How Time-Lock Encryption Works

The concept is straightforward:

You encrypt your bid with a key that doesn't exist yet. That key will only be generated by the dcipher threshold network after a specific block height is reached. Not before. The math makes early decryption impossible.

You submit your encrypted bid on-chain. Everyone can see there's an encrypted bid. Nobody can read what's inside. Not even if they control mining. Not even if they're validators. The cryptography enforces the time-lock.

When the auction closes—when the blockchain reaches the predetermined block height—the dcipher network automatically generates the decryption key. Only then can the bids be decrypted. Everyone's bids get revealed simultaneously.

No selective revealing. No gaming the system by watching other bids first. No centralized party with privileged information.

Just math enforcing fairness.

---

## How dcipher's Blocklock Makes This Real

Remember threshold signatures from the randomness examples? The same underlying infrastructure powers time-lock encryption.

For time-lock encryption, here's what happens:

![Bid Process](https://raw.githubusercontent.com/The-Web3-Compass/web3-compass-data-repository/refs/heads/main/examples/example-7/images/bid-process.png)

**1. Encryption**

You want to encrypt a bid that reveals at block 1,000,000.

You use the dcipher JavaScript library (`blocklock-js`) to encrypt your bid using a public key tied to that specific block height. This encryption is identity-based, meaning the "identity" is literally the block number.

The encrypted bid goes into your smart contract. It's stored on-chain. Visible but unreadable.

**2. Waiting**

The blockchain churns forward. Blocks get mined. Eventually block 1,000,000 arrives.

**3. Key Generation**

Now the dcipher network generates the decryption key for block 1,000,000. Multiple operators coordinate using their threshold key shares. They produce a signature that becomes the decryption key.

**4. Decryption**

An operator submits this key to your contract via a callback. Your contract decrypts the bid. Now it's readable.

This happens for all bids simultaneously because they all targeted the same block height. Perfect synchronization without any coordination between bidders.

## Why This Is Secure

The security comes from the same mathematical properties that made randomness secure:

- **No single operator can generate the key early** - Threshold coordination requires the predetermined condition to be met
- **The blockchain itself enforces the timing** - Early decryption is cryptographically impossible, not just impractical
- **No trust assumptions** - The dcipher network is decentralized. Threshold operators don't trust each other. They can't collude below the threshold. The math enforces this.

It's trustless. It's automatic. It's exactly what sealed-bid auctions needed.
