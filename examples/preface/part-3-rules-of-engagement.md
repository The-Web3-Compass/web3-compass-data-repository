# Part 3: The Rules of Engagement (Or: How This Actually Works)

Okay, you're sold. You want to build cool stuff. You're ready to dive in. But before you start slinging code like a caffeinated developer at a hackathon, let's talk about how this whole system actually works.

Because unlike most tutorials where you just read and nod along, we're actually going to make you *do things*. Build things. Submit things. Prove you learned things.

Don't worry, it's not as scary as it sounds. But there are rules. And if you ignore them, you're going to have a bad time.

---

## The Two Types of Examples

Not all examples are created equal. Some are chill reading experiences. Others are full-on coding gauntlets. Here's how to tell the difference:

### 📖 **Conceptual Examples** (The Reading Ones)

These are your "sit back and learn" examples. We walk you through concepts, explain how things work, maybe show some code, but you're not expected to build anything yourself.

Think of these like watching a cooking show. You're learning techniques, understanding ingredients, seeing the process—but you're not in the kitchen yet.

**What you do:**  
- Read the content  
- Absorb the concepts  
- Take the quiz at the end  

**What you don't do:**  
- Write code  
- Deploy contracts  
- Submit GitHub repos  

Easy, right?

---

### 💻 **Hands-On Coding Examples** (The Building Ones)

These are the real deal. You're not just reading—you're building. We give you a walkthrough, explain the concepts, and then you roll up your sleeves and actually create the thing.

These examples are marked with a special badge, so you'll know when you're about to enter the arena.

**What you do:**  
- Follow the walkthrough  
- Write the code yourself (yes, yourself—no copy-paste speedruns)  
- Deploy your contract  
- Test it to make sure it works  
- Push your code to a GitHub repo  
- Submit the repo link  

**What you don't do:**  
- Skip steps and hope for the best  
- Submit someone else's repo (we'll know, and it's awkward for everyone)  
- Rage-quit when something doesn't work (debugging is part of the learning, we promise)  

---

## The GitHub Submission System (Yes, We're Serious)

For every hands-on coding example, you'll need to submit your GitHub repository. This isn't busywork. This is proof that you actually built the thing.

### How It Works

1. **You finish building the example** – Contract written, deployed, tested. It works. You're proud. Good.

2. **You push your code to GitHub** – Create a repo, push your code, make sure it's public (or at least accessible).

3. **You submit the repo link** – At the end of the example, there's a submission form. Paste your repo link. Hit submit.

4. **We unlock the next example** – Once submitted, you can move forward. No submission? No progress. Them's the rules.

### What We're Looking For

We're not grading you on code quality or style. We're not expecting production-ready, audited, gas-optimized masterpieces.

We just want to see that you:

✅ **Actually built it** – The code is there. It compiles. It does what the example asked.

✅ **Understood the concepts** – You didn't just copy-paste blindly. You made decisions. You adapted the code to your setup.

✅ **Deployed it** – Bonus points if you include the contract address in your README, but not required.

### What Happens If You Skip This

Nothing dramatic. You just can't move forward. The next example stays locked until you submit.

Think of it like a video game checkpoint. You can't fight the boss until you've leveled up. Same deal here.

## The Quizzes (Don't Panic)

At the end of every example—whether it's conceptual or hands-on—there's a quiz.

### Why Quizzes Exist

Because reading without retention is just scrolling with extra steps.

The quizzes are there to make sure you actually absorbed the key concepts. They're not trying to trick you. They're not PhD-level cryptography exams. They're just checkpoints to confirm you're not nodding along while secretly having no idea what's happening.

### What to Expect

- **Multiple choice questions** – Usually 5-10 questions per quiz.
- **Concept-focused** – We're testing understanding, not memorization.
- **No time limits** – Take as long as you need. This isn't a speed run.
- **Retakes allowed** – Failed? No worries. Review the content and try again.

### Passing the Quiz

You need to get **at least 70%** to pass. That's it. Not asking for perfection. Just proof that you paid attention.

If you fail, you can retake it. No penalties. No shame. Just learn from the questions you got wrong and try again.

---

## The Progression System (How You Unlock Stuff)

Here's how the whole flow works:

1. **Start with the first example** – It's unlocked by default. Everyone starts here.

2. **Complete the example** – Read it, build it (if it's hands-on), take the quiz.

3. **Submit your GitHub repo** (if applicable) – Only for hands-on examples.

4. **Pass the quiz** – Get 70% or higher.

5. **Next example unlocks** – Congrats, you leveled up.

6. **Repeat** – Keep going until you've completed all the examples.

### Why This Structure?

Because learning compounds. Each example builds on the previous one. If you skip around, you'll miss context and get confused.

We're not trying to be annoying. We're trying to make sure you actually learn this stuff in a way that sticks.

---

## Tips for Success (From People Who've Been There)

### 1. **Don't Rush**

This isn't a race. Take your time. Understand the concepts. If something doesn't make sense, re-read it. Google it. Ask questions.

### 2. **Actually Build the Hands-On Examples**

Don't just read the walkthrough and think "yeah, I get it." Build it. Deploy it. Break it. Fix it. That's where the real learning happens.

### 3. **Use the Quizzes as a Learning Tool**

If you get a question wrong, don't just guess again. Go back to the content. Figure out *why* you got it wrong. That's the valuable part.

### 4. **Keep Your Repos Organized**

You're going to build multiple projects. Keep them organized. Future you will thank present you.

### 5. **Don't Be Afraid to Experiment**

The examples give you a foundation, but feel free to tweak things. Add features. Try different approaches. Break stuff on purpose to see what happens. That's how you go from "following a tutorial" to "actually understanding the tech."
