# The Rules of Engagement (Or: How This Actually Works)

You're sold. You want to build. But before you start slinging code, let's talk about how this system works.

Unlike most tutorials where you just read and nod, we're making you *do things*. Build things. Submit things. Prove you learned things.

Not scary. But there are rules. Ignore them, have a bad time.

---

## The Two Types of Examples

**Conceptual Examples**: Sit back and learn. We walk through concepts, explain how things work, maybe show code. You're not building anything yourself.

What you do: Read, absorb, take the quiz. What you don't do: Write code, deploy contracts, submit GitHub repos. Easy.

**Hands-On Coding Examples**: The real deal. You're building. We give you a walkthrough, you roll up your sleeves and create the thing. These have a special badge.

What you do: Follow walkthrough, write code yourself (no copy-paste speedruns), deploy contract, test it, push to GitHub, submit repo link.

What you don't do: Skip steps, submit someone else's repo (we'll know, it's awkward), or rage-quit when something breaks. Debugging is learning.

---

## The GitHub Submission System

For every hands-on example, you submit your GitHub repo. This is proof you built it.

Finish building. Contract written, deployed, tested. Works. You're proud. Good. Push to GitHub—create repo, push code, make it public. At the end of the example, paste your repo link in the submission form. Hit submit.

No submission? No progress. Them's the rules.

**What we're looking for**: Not code quality or style. Not production-ready, audited, gas-optimized masterpieces. Just proof you built it—code compiles, does what the example asked, you understood concepts and made decisions. Bonus points for including contract address in README, not required.

---

## The Quizzes

Every example ends with a quiz. Why? Because reading without retention is scrolling with extra steps.

**What to expect**: 5-10 multiple choice questions testing understanding, not memorization. No time limits. Not a speed run.

**Passing**: Need 70%. Not perfection. Just proof you paid attention.

**Fail?** Review content, try again. Retakes allowed. No penalties. No shame.

---

## Tips for Success

**Don't rush**. Not a race. Understand concepts. Something doesn't make sense? Re-read. Google. Ask questions.

**Actually build** hands-on examples. Don't just read and think "yeah, I get it." Build. Deploy. Break. Fix. That's where learning happens.

**Use quizzes as learning tools**. Got a question wrong? Don't just guess again. Go back. Figure out why. That's the valuable part.

**Keep repos organized**. You're building multiple projects. Future you will thank present you.

**Experiment**. Examples give you foundation. Tweak things. Add features. Try different approaches. Break stuff on purpose. That's how you go from "following tutorials" to "understanding the tech."
